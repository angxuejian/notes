# 优化 JavaScript：为了乐趣，也为了收益

> [原文地址](https://romgrk.com/posts/optimizing-javascript)，原文章中的示例并不会搬运过来，这里只作为翻译正文。

我经常觉得，很多 JavaScript 代码其实都比它本可以达到的速度慢得多，只是因为没有采用更合理的优化方式。下面总结的是一些我在实践中认为非常实用的优化技巧。

需要注意的是，性能和可读性往往需要权衡。至于什么时候应该优先追求性能，什么时候应该优先考虑可读性，就留给读者自己判断了。

另外还要说明一点，只要谈到性能优化，就绕不开基准测试。

如果这个函数本来只占整个程序运行时间的一小部分，那么花几个小时把它优化到快上 100 倍，也是没有意义的

如果要进行性能优化，第一步，也是最重要的一步，就是进行基准测试。关于这一点，我会在后面的章节中详细介绍。

也请注意，微基准测试通常都有一定的局限性，这一点也包括本文中的这些测试。我已经尽力避免常见问题，在采用文中的优化方法之前，请一定要先做好基准测试。

凡是能够提供可运行示例的场景，我都附上了对应的示例。这些示例默认展示的是我在自己的机器（Brave 122，Arch Linux）上运行得到的结果，不过你也可以亲自在自己的环境中运行它们。尽管我很讨厌这样说，Firefox 在性能优化方面稍显落后，不过 [目前](https://www.mozillafoundation.org/en/?form=donate-header) Firefox的访问量也是非常少，因此，我不建议把在 Firefox 上得到的测试结果作为判断性能的参考。
 
## 0. 避免不必要的工作

这听起来可能很明显，但仍然有必要把它放在这里，因为性能优化的第一步永远都是这一点：如果你想进行优化，首先应该思考能否减少需要执行的工作。这包括缓存计算结果、惰性计算以及增量计算等技术。具体采用哪种方式取决于实际场景。例如，在 React中，这通常意味着使用 `memo()`、`useMemo()` 以及其他适用的内置优化机制。

## 1. 避免字符串比较

JavaScript 很容易让开发者忽略字符串比较背后的真实开销。如果是在 C 语言中比较字符串，你通常会调用 `strcmp(a, b)` 函数；而 JavaScript 则直接使用 `===`，因此你看不到 `strcmp` 的存在。但实际上，底层依然需要进行字符串比较。字符串比较通常（尽管并非总是如此）需要逐个比较两个字符串对应位置上的字符，因此它的时间复杂度通常为 O(n)。一种常见但应尽量避免的 JavaScript 写法，是使用字符串来表示枚举值。随着 TypeScript 的出现，这种写法已经很容易避免，因为 TypeScript 的 enum 默认使用整数值。

> 关于基准测试
>
> 百分比的计算方式是：某项测试在 1 秒内完成的操作数 ÷ 表现最佳测试用例在 1 秒内完成的操作数。因此，表现最好的测试结果为 100%，其他结果则按比例计算；百分比越高越好。


正如你所见，两者之间的性能差距可能相当明显。这种差距未必来自 `strcmp` 逐字符比较字符串的开销，因为 JavaScript 引擎有时会使用字符串池，只需比较字符串引用即可，而无需逐字符比较。另一方面，这种差距还与 JS 引擎的内部表示有关：整数通常按值（passed by value）传递，而字符串始终以指针（passed as pointers）的形式传递；访问内存本身就是一项代价较高的操作（见第 5 节）。因此，在大量使用字符串的代码中，这种差异可能会带来非常显著的性能影响。

举个实际例子，我仅通过将字符串常量替换为数字，就让[ JSON5 javascript](https://github.com/json5/json5/pull/278) 解析器的运行速度提高了 2 倍*。

> \* 可惜的是，这个优化最终没有被合并。不过，开源项目就是这样
